# Ray Growth OS 使用指南

[返回中文 README](../README.zh-CN.md) | [English guide](USER_GUIDE.md) | [English README](../README.md)

这是一份给第一次使用 Ray Growth OS 的指南。照着「首次跑通」完成一轮，你就会知道它如何从 X 的公开讨论中找到值得互动的人、准备草稿，并把执行结果用于下一轮改进。

## 先用一句话理解它

Ray Growth OS 不是自动发帖或自动私信工具。

它帮助你把分散的公开讨论整理成一个人工可控的增长流程：**先定义目标 → 找到讨论 → 判断优先级 → 准备草稿 → 由你亲自互动 → 记录结果 → 改进下一轮。**

你可以把它理解为：面向 X 公开讨论的「找人 + 互动决策 + 复盘」工作台。

## 5 分钟首次跑通

第一次不需要配置任何 API。建议先走手动流程，确认它适合你的场景后，再配置自动查询和 AI。

### 1. 打开工作台

本地启动项目后，访问 `http://localhost:3001`。

左侧的主流程依次是：

| 页面 | 你在这里做什么 | 什么时候用 |
| --- | --- | --- |
| 总览 | 查看今日队列和下一步 | 想知道现在该做什么时。 |
| 定位找人 | 描述目标，生成搜索 Prompt，导入讨论 | 每一轮开始时。 |
| 互动队列 | 排序、看草稿、打开原帖、记录结果 | 日常执行时。 |
| 竞品洞察 | 从竞品、KOL 或社区账号周围找讨论 | 想换一个切入角度时，可选。 |
| 设置 | 配置 Grok、AI 模型和公开 X 主页 | 想启用自动能力时，可选。 |

### 2. 填写你的定位

进入**定位找人**，先填写左侧表单。无需写得很长，但要具体。

建议优先写清楚这五项：

| 字段 | 要回答的问题 | 示例 |
| --- | --- | --- |
| 账号 / 产品名称 | 你是谁或你在做什么？ | `API Observe` |
| 增长目标 | 你希望通过互动获得什么？ | `找到早期 SaaS 技术负责人，验证可观测性痛点` |
| 定位描述 | 你提供什么价值？ | `帮助小团队发现 API 故障和慢请求` |
| 目标人群 | 你想影响谁？ | `刚上线 SaaS 的技术负责人、后端开发者` |
| 主题 / 痛点 | 他们会讨论什么问题？ | `API 报错、告警噪音、排查慢、可观测性成本` |

下面的**互动目的 / 下一步**和**产品 / 身份露出方式**会影响 AI 草稿。用它告诉系统：

- 这次互动想达成什么，例如「先分享一个排查思路，再邀请继续交流」；
- 什么时候可以提产品，哪些承诺不能写。

不要写成广告文案。越像真实的使用场景和问题，搜索与草稿就越有用。

### 3. 用 Grok 找公开讨论

右侧会自动生成一份 Grok Prompt。选择两种方式之一：

#### 方式 A：手动搜索（推荐第一次使用）

1. 点击**复制 Prompt 并打开 Grok**。
2. 在 Grok 执行搜索。
3. 将结果复制回来，建议每条一行，使用下面格式：

```text
X | 作者或账号 | 帖子链接 | 一句话摘要 + 为什么值得处理
```

4. 在工作台预览结果，确认后点击**导入结果**。

你不需要提供浏览器 Cookie 或 X 登录信息给工作台。

#### 方式 B：自动查询（配置后使用）

1. 进入**设置**，保存 Grok/codeproxy 的 API 密钥和模型。
2. 返回**定位找人**，选择**一键自动查询**。
3. 工作台会把当前 Prompt 提交给中转，解析为待确认的结构化信号。
4. 仍然先审核，再导入。

自动查询省的是复制粘贴，不代替人工判断。链接、内容真实性和是否适合互动，都应在导入前确认。

### 4. 在互动队列决定要做什么

导入后，去**互动队列**。每一条讨论都包含：来源、分数、推荐动作、原因和草稿。

建议按下面方式处理：

1. 先看分数高、痛点具体、仍然新鲜的条目。
2. 打开原帖，确认上下文和作者确实相关。
3. 选择一个草稿作为起点，按照你的真实经验改写。
4. 由你亲自在 X 发送回复、引用或后续私下沟通。
5. 回到队列标记处理状态。

草稿的含义：

| 草稿 | 用途 |
| --- | --- |
| 直接回复 | 适合直接回到原帖下。 |
| 引用转发 | 将讨论延展为自己的公开观点。 |
| 内容选题 | 留作后续原创帖、长帖或文章的素材。 |
| 私下跟进 | 仅在对方明确有需求、适合继续交流时使用。 |

系统不会自动替你发内容。每条草稿都应审核，尤其要删除无法证明的结论、承诺、链接或过度销售的话术。

### 5. 记录真实结果

处理一条后，在队列中标记：已回复、已引用、已收藏、暂缓或跳过。之后再补充反馈，例如：

- 有回复；
- 无回复；
- 被关注；
- 被转发。

这一步很重要。它让下一轮不只是“再找一些帖子”，而是根据实际结果调整优先级和写法。

### 6. 生成增长记忆

积累了一些已处理且有反馈的条目后，在互动队列的增长记忆区域点击**生成增长记忆**。

它会提炼出：

- 哪些关键词和痛点更值得优先处理；
- 哪些类型的信号常常没有后续；
- 哪些回复方式可以复用，哪些需要避免；
- 下一轮可以验证的一个具体实验。

先阅读，再决定是否点击**应用增长记忆**。它是可暂停、可清空的优先级参考，不会永久修改原始数据。

以后有新反馈时，点击**增长到下一轮**。系统只学习尚未处理过的新反馈，并复核相关旧规则：相似规则会合并，得到支持的规则会增强，出现反证的规则会降级或暂停。免费版最多启用 10 条规则，完整历史不会全部放进 Prompt，因此不会随着轮数增加而无限消耗上下文。

## 可选能力

### AI 评分和草稿

在**设置**中配置 AI Responses 兼容的密钥和模型后，可以使用：

- AI 帮填定位；
- AI 语义评分；
- AI 草稿生成；
- AI 增长记忆。

AI 草稿会逐条跟随原帖语言生成；即使一次选择了中、英、日等不同语言的帖子，每条草稿也会分别匹配自己的原帖语言。新查询结果会保存原帖语言的精简文本和一个短语言标记（如 `en`）；只有两者都无法判断时，才使用当前界面语言兜底。重新导入相同链接可以为旧记录补全这些信息，不会覆盖已保存的处理状态和反馈。

未配置时，工作台仍会保留本地规则的排序和基础草稿，因此可以先不用 API 测流程。

### 竞品洞察

当你不知道该搜什么，或想从成熟账号的受众周围切入时，使用**竞品洞察**：

1. 输入一个公开 X 账号地址，可以是竞品、行业 KOL、社区或理想用户。
2. 点击**分析账号并生成线索**。
3. 查看它与你的定位之间的受众重叠、机会缺口和推荐角度。
4. 只导入外部受众、评论者和相关讨论；目标账号本人不应成为互动信号。

它只处理公开信息，不读取私信、后台数据或非公开内容。

### X Helper Chrome 插件

插件用于减少“我回复后，到底有没有结果”的手动记录工作。它是可选的。

安装：

1. 保持工作台页面打开：`http://localhost:3001` 或 `http://127.0.0.1:3001`。
2. 在 Chrome 打开 `chrome://extensions/`，开启开发者模式。
3. 点击**加载已解压的扩展程序**，选择 `extension/ray-growth-os-x-helper` 目录。
4. 在 App 设置里保存公开 X 主页，或者在插件弹窗里保存一次 X 用户名；弹窗不需要保持打开。
5. 从工作台的**复制回复并打开原帖**进入 X。插件会自动读取并关联当前条目，不需要手动读取队列。
6. 由你亲自发布回复；回复显示在当前原帖对话页后，插件会自动保存回复链接并回写状态。
7. 若漏抓，停留在原帖对话页，点击**找回当前页回复并回写**。
8. 已经保存回复链接后，才使用**巡检已记录的回复链接**检查后续公开反馈。

注意：“巡检已记录的回复链接”不能找回一次从未保存过的回复；这种情况应先使用“找回当前页回复并回写”。项目更新后需要在 `chrome://extensions/` 重新加载插件，并刷新已打开的 App/X 页面。

插件不会替你发送内容、不读私信、不绕过登录，也不调用 X 付费 API。详细说明见[插件 README](../extension/ray-growth-os-x-helper/README.md)。

## 哪些配置是必须的？

| 你的目标 | 是否需要配置密钥 | 最小做法 |
| --- | --- | --- |
| 先验证流程 | 不需要 | 手动复制 Prompt 到 Grok，再粘贴结果。 |
| 自动查询 Grok | 需要 Grok/codeproxy 密钥 | 在设置中配置 Grok。 |
| AI 评分、草稿、定位建议 | 需要 AI 模型密钥 | 在设置中配置 AI。 |
| 自动回流 X 公开反馈 | 不需要 API 密钥，但需安装 Chrome 插件 | 安装 X Helper。 |

API 设置与工作台状态保存在这台电脑的本地 SQLite 数据库中，同一台电脑上的浏览器会读取同一份数据。API 密钥不会导出到工作台 JSON 备份。首次升级只迁移设置，旧工作台记录会被刻意留在原浏览器、不带入新数据库，以便从空白状态测试。部署给团队前，请改为服务端密钥管理。

## 常见问题

### 我没有 X 账号，能用吗？

可以使用导入、评分、草稿和复盘流程；但核心场景是分析 X 公开讨论，实际互动和插件反馈功能需要 X。

### 为什么搜索结果不相关？

先改定位，不要只加关键词。目标人群、具体痛点和你能贡献的价值越清晰，结果越好。再在展开的 Prompt 中补充少量关键词微调。

### 为什么导入后没有可处理条目？

可能是结果重复、缺少可用内容，或是自己的账号/产品账号被排除了。尝试换更具体的痛点，或在竞品洞察中选择一个受众更接近的公开账号。

### AI 草稿可以直接发送吗？

不建议。将它作为初稿，补上你真实的判断与经验，并删除不准确、过度承诺或与原帖不相关的内容。

### 数据会上传到哪里？

工作台状态保存在这台电脑的本地 SQLite 数据库。使用自动 Grok 查询或 AI 功能时，当前请求内容会发送到你配置的 codeproxy/模型服务；请不要放入私信、密钥或非公开客户数据。

### 如何迁移或备份数据？

在**定位找人**页底部使用 JSON 备份/恢复。备份包含工作台数据与草稿，不包含 API 密钥和 X 主页设置。

## 推荐的日常节奏

1. 每周更新一次定位、目标人群和痛点关键词。
2. 每次导入一小批高质量讨论，而不是追求数量。
3. 每天处理少量最相关的条目，优先做真实、有价值的回复。
4. 每周回看反馈，生成并审核增长记忆。
5. 根据有效/无效模式调整下一轮定位和搜索条件。

如果你愿意把一个完整循环跑通，Ray Growth OS 的价值不在于“多生成几条文案”，而在于让你逐渐知道：**谁值得互动、说什么更有用、以及下一轮该去哪里找人。**
